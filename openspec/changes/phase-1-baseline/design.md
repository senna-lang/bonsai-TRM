## Context

Phase 0 で Bonsai × AppWorld の統合が確認され、パイロット評価でスコープが決定された。本 Phase では確定スコープでベースライン測定を行い、スキャフォールド適用前の基準値と失敗パターンを確立する。

## Goals / Non-Goals

**Goals:**
- 確定スコープの全タスクでベースライン成績(タスク目標達成率)を測定
- 失敗モードを論文と同じ 4 カテゴリで分類
- チェックポイント機構でセッション切断に対応
- 推論時間の統計を収集

**Non-Goals:**
- スキャフォールドの実装(Phase 2)
- Qwen3-8B AWQ の再現(参考値として論文数値を引用するのみ)
- 記事執筆(Phase 4)

## Decisions

### 1. チェックポイントは completed_results ベースで実装
**選択**: タスク完了ごとに結果を Drive に保存し、再開時はスキップ
**理由**: Colab Pro でもセッション切断の可能性あり。シンプルなスキップロジックで十分。

### 2. 失敗モード分析は手動分類をベースとする
**選択**: GPT-4o 自動分類(論文と同じ)を第一候補、コスト・手間が問題なら手動
**理由**: 論文と同じ手法で比較可能性を確保。Bonsai のタスク数は最大 168 で手動でも可能な範囲。

### 3. ログは全ターンの LLM 入出力を保存
**選択**: 各ターンの prompt / completion / observation を JSON Lines で保存
**理由**: 失敗モード分析と Phase 2 のスキャフォールド設計に必要。ストレージは十分。

## Risks / Trade-offs

- **[ベースライン 0%]** → Mistral 7B が 0% なので Bonsai も 0% の可能性あり。ただし 0% でもスキャフォールドで改善が出るかが本題なので問題なし
- **[セッション切断]** → チェックポイント機構で対応。最悪の場合は複数セッションに分割
- **[推論時間が予想超過]** → Phase 0 の実測に基づきスコープを調整済みなのでリスク低
